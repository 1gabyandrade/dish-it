# Dish-It Triage Log

This log records the main technical hurdles encountered while building Dish-It
and how they were solved during development.

## 1. Navbar Active State

### Problem

The `Home` item in the navbar needed to link back to the main page and appear
underlined only when the user was on the home page.

### Cause

The navbar was originally static HTML. The `Home` item was just a `<span>`, so
it could not behave like a real link or reflect the active page.

### Solution

I created a reusable navbar function that accepts an `active_page` value. The
navbar now builds real links and applies an `is-active` class only to the
current page.

### Result

The active page is visually clear, and the navbar is ready for future pages.

## 2. User Logged Out After Refresh

### Problem

After logging in, refreshing the page or reopening the app logged the user out.

### Cause

The app relied only on Streamlit's `st.session_state`, which is tied to a single
session and does not persist across reloads.

### Solution

I introduced a temporary auth token stored in memory using `st.cache_resource`
and passed it through URL query parameters. On app load, the token is read and
used to restore the logged-in user.

### Result

The user stays logged in after refreshing the page or reopening the app, as long
as the Streamlit server is running.

## 3. Saving Favorite Recipes

### Problem

Users needed a way to save generated recipes and view them later in `My Recipes`.

### Cause

Recipes were only stored temporarily in session state. There was no database
table or service layer for saved recipes.

### Solution

I added a `favorite_recipes` table and a `favorite_service.py` service. Each
saved recipe stores the user, recipe title, full recipe text, ingredients, and a
hash to avoid duplicate saves.

### Result

Users can save recipes, see them in `My Recipes`, and remove them later.

## 4. Recipe History

### Problem

The app needed a `History` page showing every recipe generated by the user.

### Cause

Generated recipes were only shown on the home page and were not persisted unless
the user favorited them.

### Solution

I added a `recipe_history` table and a `history_service.py` service. When a
valid recipe is generated, it is automatically saved to history.

### Result

The `History` page now shows all generated recipes for the logged-in user.

## 5. New Service Import Issue

### Problem

After adding `history_service.py`, Streamlit showed:

```text
ModuleNotFoundError: No module named 'services.history_service'
```

### Cause

The Streamlit server was still running with an older import state, and the
`services` folder did not explicitly declare itself as a Python package.

### Solution

I added `services/__init__.py` and restarted the app. I also verified the import
directly from the terminal.

### Result

The import worked correctly and the history feature loaded.

## 6. My Account Button Styling

### Problem

The `My Account` page buttons had different widths and visual weights. The
confirmation buttons looked inconsistent with `Save Changes` and `Delete
Account`.

### Cause

Each button had its own CSS rule with different `min-width`, `height`, and font
weight values.

### Solution

I consolidated the account button CSS into one shared rule for sizing. Then I
kept separate color rules for normal actions and destructive actions.

### Result

All account buttons now share the same size and shape. Yellow buttons and red
buttons keep their intended colors.

## 7. Outdated Documentation

### Problem

The original README no longer matched the app. It mentioned Gemini and mock
recipes, but the app now uses OpenRouter, SQLite, favorites, history, and account
management.

### Cause

The product changed faster than the documentation.

### Solution

I rewrote the README as a Founder's Manual with current setup instructions,
project structure, environment variables, database details, and page
descriptions.

### Result

The documentation now reflects the real project and can be used to run or review
the app.

## 8. Unused Code and Duplicated Helpers

### Problem

Some files and helpers were no longer needed, and ingredient encoding logic was
duplicated between services.

### Cause

The app evolved from a smaller MVP into multiple pages and services.

### Solution

I removed the unused Gemini service, cleaned up OpenRouter error handling, and
created `recipe_record_utils.py` for shared recipe record helpers.

### Result

The codebase is easier to read and has less duplicated logic.

## 9. URL Token Tradeoff for Session Persistence

### Problem

The initial implementation used auth tokens in the URL to persist sessions.
However, URL tokens can be copied and reused in another browser, which creates a
potential security issue.

### Cause

Any value placed in the URL is inherently shareable. This makes URL-based tokens
unsuitable for production authentication.

### Solution

I temporarily removed URL tokens and switched to `st.session_state` navigation
for better security. However, after discussion with the professor, I reintroduced
URL tokens specifically for debugging and development purposes.

### Result

The app now uses URL tokens to restore sessions across reloads during
development, making debugging easier. In a production environment, this would be
replaced with secure cookies and server-side session management.

## 10. Backend Validation and Error Handling

### Problem

User input and API errors were not clearly validated or distinguished. Invalid
inputs and backend failures could produce misleading messages.

### Cause

Validation logic was minimal, and API failures were treated the same as normal
“no recipe found” cases.

### Solution

I added input normalization and validation for usernames, emails, and passwords.
I also improved API handling by distinguishing between OpenRouter failures and
valid “no recipe found” responses.

### Result

The app now provides clearer feedback, maintains cleaner data in the database,
and behaves more predictably under error conditions.

## 11. Ingredient Data Normalization

### Problem

Ingredient data could be stored with inconsistent formatting (e.g., casing,
whitespace, duplicates).

### Cause

Ingredients were taken directly from UI input without normalization before
saving.

### Solution

I added normalization logic to ensure ingredients are trimmed, lowercased, and
deduplicated before being stored in the database.

### Result

Stored data is more consistent, reducing duplication and improving overall data
quality across history and favorites.
